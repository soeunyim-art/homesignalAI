## 2. 모델 개선 사항 (predict_model.py)

### 2-1. 학습/평가 분할 방식 변경

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 방식 | TimeSeriesSplit(5-fold) 교차검증 | 시계열 7:3 고정 분할 |
| 학습셋 | 전체 rolling | 앞 70% (2020-01 ~ 약 2024-07) |
| 평가셋 | 없음 (고정 test set 없음) | 뒤 30% (약 2024-08 ~ 2026-01) |
| 최종 학습 | - | 전체 데이터로 재학습 (예측에 사용) |

**이유:** 고정된 test set 없이 cross-validation만 하면 미래 데이터 누수 가능성 + 발표 시 명확한 평가 지표 제시 불가

### 2-2. 아파트별 예측 기능 추가 (방식 A)

**구조:**
```
동별 Ridge 모델 → 동별 예측 변화율(%) 산출
                         ↓
             apt_trade 최근 6개월 평균 거래가 조회
                         ↓
          (동별 변화율) × (아파트 현재가) = 아파트별 예측가
```

**추가된 함수:**
- `fetch_apt_current_prices()` — apt_trade에서 아파트별 최근 6개월 평균가 조회 (2건+ 거래 기준)
- `predict_apt_level()` — 동별 변화율을 아파트별로 적용
- `save_apt_predictions_to_supabase()` — predictions_apt 테이블 저장

**출력 추가:**
- `prediction_result_apt.csv` — 아파트별 1/2/3개월 예측가
- Supabase `predictions_apt` 테이블 저장

## 3. DB 스키마 정규화 (FK 설계)

### 설계 원칙
- **raw 수집 테이블** (apt_trade, apt_rent): FK 미적용 → 수집 중 신규 동명 등장 시 오류 방지
- **집계/예측 테이블** (predictions, predictions_apt): FK 적용 → 데이터 무결성 보장

### 최종 테이블 구조

```
dongs (PK: dong_name, 컬럼: gu_name, dong_code)
  └── apartments.dong → FK
        └── predictions_apt.(dong, apt_name, area) → FK
  └── predictions.dong → FK

apt_trade   ─ FK 없음 (raw)
apt_rent    ─ FK 없음 (raw)
interest_rate ─ 독립 테이블
```

### 신규 테이블

| 테이블 | 역할 |
|--------|------|
| `dongs` | 동 마스터 (dong_name PK, gu_name, dong_code) |
| `apartments` | 아파트 마스터 (dong+apt_name+area UNIQUE, FK→dongs) |
| `predictions` | 동별 예측 (FK→dongs) |
| `predictions_apt` | 아파트별 예측 (FK→dongs, FK→apartments) |

### FK를 만들어야 하는 이유
1. **데이터 무결성** — 존재하지 않는 동명으로 예측 저장 차단
2. **고아 데이터 방지** — 아파트 마스터 없는 예측 데이터 방지
3. **Supabase 자동 관계 인식** — Table Editor 관계 다이어그램, nested query 지원
4. **프론트 연동 안전성** — JOIN 쿼리 신뢰성 보장

## 5. R² KPI 목표

| 단계 | 예상 R² | 기준 |
|------|---------|------|
| 이전 (동대문구만) | 0.15~0.26 | 달성 |
| **MVP 목표** | **≥ 0.40** | **현재 단계** |
| 고도화 목표 | ≥ 0.55 | 피처 추가 후 |

> 부동산 월간 변화율 예측에서 R² 0.40 이상 = 실용적 수준
> (학계 기준: 0.30 이상 유의미, 0.50 이상 우수)
