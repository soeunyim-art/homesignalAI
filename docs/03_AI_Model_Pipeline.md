# HomeSignal AI — 시계열 모델 파이프라인 설계서

## 1. 개요 (Overview)

본 문서는 동대문구 부동산 매매지수 예측을 위한 시계열 모델의 **수집-전처리-학습-평가** 파이프라인을 정의합니다. 수치 데이터와 비정형 뉴스 데이터를 결합하여 예측의 정교함을 높이는 데 목적이 있습니다.

---

## 2. 데이터 파이프라인 (Data Pipeline)

### 2.1. 데이터 추출 (Data Extraction)

| 유형 | 소스 |
|------|------|
| **수치 데이터** | 국토교통부 실거래가 API, 한국부동산원 매매가격지수, 한국은행 기준금리 |
| **비정형 데이터** | 구글 뉴스 및 부동산 커뮤니티 크롤링 데이터 |

### 2.2. 데이터 전처리 (Data Preprocessing)

| 항목 | 처리 방법 |
|------|----------|
| **결측치 처리** | 부동산 거래 비수기나 데이터 누락 구간에 대한 보간법(Interpolation) 적용 |
| **이상치 제거** | 신고가 취소 건 및 증여성 특수거래 데이터 필터링 |
| **정규화(Normalization)** | 서로 다른 단위(금리 vs 매매가)를 가진 변수들을 동일한 스케일로 조정 |

---

## 3. 탐색적 데이터 분석 (EDA)

- **트렌드 분석:** 동대문구 매매지수의 장기 추세 및 계절성(이사철) 파악
- **상관관계 분석:** 기준금리 변동과 실거래가 반등 사이의 시차(Time Lag) 분석
- **시각화:** 변수 간의 관계를 히트맵 및 산점도로 시각화하여 유의미한 변수 식별

---

## 4. 피처 엔지니어링 (Feature Engineering)

HomeSignal AI만의 차별화된 **'이벤트 기반 신호'**를 변수화합니다.

| 피처 유형 | 설명 |
|----------|------|
| **뉴스 키워드 빈도** | 'GTX', '청량리 재개발', '이문휘경뉴타운' 등 핵심 키워드의 노출 빈도를 시계열 변수로 변환 |
| **이벤트 더미 변수** | 대규모 분양, 전매 제한 해제, 금리 동결 등 주요 정책 발표 시점을 0과 1로 코딩 |
| **계절성(시즌) 더미** | 개학(2~3월·8~9월), 이사시즌(1~2월·12월), 결혼시즌(5월·10월) 등 임대·수요 패턴 반영 (상세: [§9](#9-계절성-피처-및-임대율-패턴-적용-계획)) |
| **이동 평균(Moving Average)** | 단기(5주) 및 장기(20주) 매매가 이동 평균을 통해 추세 반전 신호 포착 |

---

## 5. 모델링 및 학습 (Modeling & Training)

### 5.1. 알고리즘 선정

| 모델 | 역할 |
|------|------|
| **Prophet** | 전체적인 시장 트렌드와 계절성 파악을 위한 베이스라인 모델 |
| **LightGBM** | 뉴스 이슈 및 정책 변수의 가중치를 학습하기 위한 머신러닝 모델 |
| **Ensemble** | 두 모델의 결과값을 가중 평균하여 최종 예측치 도출 |

### 5.2. 학습 프로세스

- **다양한 분석 모형 활용:** 여러 알고리즘을 동시에 학습시켜 최적의 성능을 보이는 조합 탐색
- **Hyper Parameter 튜닝:** 학습률(Learning Rate), 은닉층 수, 트리의 깊이 등을 경험적으로 최적화

---

## 6. 평가 및 검증 (Evaluation & Validation)

| 항목 | 내용 |
|------|------|
| **평가 KPI** | RMSE(Root Mean Square Error), MAE(Mean Absolute Error)를 통해 예측 오차 측정 |
| **Backtesting** | 과거 특정 시점 데이터를 제외하고 학습시킨 후, 실제 발생한 가격과 비교 검증 |
| **최종 모델 선정** | 검증 지표가 가장 우수한 모델을 실시간 서비스용으로 확정 |

---

## 7. 모델 적용 및 모니터링 (Deployment & Monitoring)

| 모드 | 설명 |
|------|------|
| **Batch** | 주기적으로(예: 매주 월요일) 전체 데이터를 재학습하여 모델 업데이트 |
| **Real Time** | 사용자 요청 시 저장된 모델을 통해 즉시 예측치 반환 |
| **성능 모니터링** | 실제 실거래가 발표 시점마다 모델의 예측치와 대조하여 성능 저하 여부 감시 |

## 8. 파이프라인 단계별 산출물

| 단계 | 산출물 | 후속 작업 |
|------|--------|----------|
| 데이터 추출 | Raw CSV/JSON, DB 적재 | 전처리 스크립트 |
| 전처리 | 정제된 시계열 테이블 | EDA 노트북 |
| EDA | 분석 리포트, 변수 선정 목록 | 피처 엔지니어링 |
| 피처 엔지니어링 | 피처 테이블 | 학습 스크립트 |
| 학습 | 모델 파일(.pkl, .joblib) | 평가 |
| 평가 | RMSE/MAE 리포트 | 모델 선정 |
| 배포 | API 연동, 배치 스케줄 | 모니터링 |

## 9. 계절성 피처 및 임대율 패턴 적용 계획

개학·이사시즌·결혼시즌으로 **임대(전세/월세) 수요 패턴**을 변수화하고, 이를 부동산 시계열 예측 모델에 반영하는 계획입니다.

### 9.1 적용 방향

| 방향 | 설명 | 필요 데이터 |
|------|------|-------------|
| **A. 매매 모델의 설명 변수** | 개학·이사·결혼 시즌을 달력 기반 더미로 넣어 매매가·거래량의 계절성(이사철 등)을 설명 | 매매 시계열 + 시즌 정의(날짜) |
| **B. 임대 전용 예측 모델** | 전세/월세 시계열을 별도 예측하고, 시즌 더미를 피처로 활용 | 국토교통부 전월세 실거래가 API |

현재 설계의 **이벤트 더미 변수** 및 **계절성(이사철)**을 확장하여 위 시즌을 명시적으로 반영합니다.

### 9.2 시즌 정의 (한국 기준)

| 시즌 | 일반적 시기 | 활용 목적 |
|------|-------------|-----------|
| **개학** | 2~3월, 8~9월 | 학군·입시 수요 → 전세/월세·이사 수요 증가 |
| **이사시즌** | 1~2월, 12월 | 이사 수요 집중 → 매매·전세 거래량·가격 변동 |
| **결혼시즌** | 5월, 10월 | 신혼 수요 → 전세/소형 매매 수요 변동 |

- 구체적 주/일 범위는 EDA 또는 도메인 전문가 검토 후 확정.
- 필요 시 **강도 변수**(0~1 또는 빈도 가중치)로 확장 가능.

### 9.3 피처 추가 사항

| 피처 유형 | 변수 예시 | Prophet | LightGBM |
|-----------|-----------|---------|----------|
| **시즌 더미** | `is_school_season`, `is_moving_season`, `is_wedding_season` (0/1) | `add_regressor()` 로 추가 | 범주/수치 피처로 추가 |
| **연·월·주** | `month`, `week_of_year` | 연간 계절성에 포함 | 계절성 보조 피처 |

- **Prophet:** 연간 계절성만으로는 개학/이사/결혼 구간이 명확히 나뉘지 않을 수 있으므로, `add_regressor()` 로 시즌 더미를 명시적으로 추가.
- **LightGBM:** 기존 이벤트 더미와 동일한 형태로 시즌 더미 추가하여 학습.

### 9.4 임대율 패턴 적용

- **매매 모델에만 적용 시:** 개학·이사·결혼 시즌을 “임대/이사 수요가 높은 구간”의 **프록시**로 사용. 임대 시계열 데이터 없이 달력 기반 더미만으로 매매 계절성 보정 가능.
- **임대(전세/월세) 예측 추가 시:** 국토교통부 **전월세 실거래가 API** 수집 후, 동일한 시즌 더미를 임대 시계열 모델의 피처로 사용하여 임대율·임대가 패턴 예측.

### 9.5 구현 시 참고

- 시즌 구간 정의는 설정 파일(예: `config/season_definitions.yaml`) 또는 상수 모듈로 관리하여 재학습·A/B 테스트 시 조정 용이하게 할 것.
- EDA 단계에서 **계절성(이사철)** 분석 시 개학·이사·결혼 구간별 거래량/가격 차이를 시각화하여 시즌 정의를 검증할 것.
