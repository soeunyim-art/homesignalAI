## 1. 개요

본 문서는 HomeSignal AI 프로젝트의 **역할별 업무분장**을 정의합니다. 각 담당자의 책임 범위, 산출물, 협업 포인트를 명시하여 원활한 개발 진행을 목표로 합니다.

## 2. 역할별 업무분장 요약

| 역할 | 담당 범위 | 핵심 산출물 |
|------|-----------|-------------|
| **개발1** | 국토교통부 API 데이터 수집 + Supabase + 기획 업무 | 실거래가 수집 파이프라인, Supabase 프로젝트·테이블·데이터 적재, PRD/로드맵 관리 |

## 3. 개발1 — 데이터 수집 및 기획 담당

### 3.1 역할 정의

국토교통부 실거래가 API를 활용한 **데이터 수집 파이프라인** 구축, **Supabase DB 운영**, 프로젝트의 **기획 업무**를 담당합니다.

### 3.2 담당 업무 상세

#### A. 국토교통부 API 데이터 수집

| # | 작업 | 상세 |
|---|------|------|
| 1 | 국토교통부 실거래가 API 연동 | Open API 인증, 요청 형식·응답 파싱 구현 |
| 2 | 동대문구 데이터 수집 | 시·군·구 코드 파악, 월별/지역별 데이터 조회 |
| 3 | 전월세 데이터 수집 (선택) | 국토교통부 전월세 실거래가 API 연동 |
| 4 | 데이터 전처리 | 결측치, 이상치 처리, `houses_data` 스키마에 맞는 변환 |
| 5 | 배치 스케줄 설계 | 주기적 수집(예: 월 1회) 및 Supabase 적재 로직 |
| 6 | 개발2와 협의 | 적재 테이블 구조, 컬럼 매핑, 데이터 형식 합의 |

#### B. Supabase DB 운영

| # | 작업 | 상세 |
|---|------|------|
| 1 | Supabase 프로젝트 생성 | PostgreSQL 인스턴스, pgvector 확장 활성화 |
| 2 | 4개 테이블 생성 | `db_Schema(Sql)_mokdb.md` DDL 실행 |
| 3 | 인덱스·RLS 설정 | `contract_date`, `keywords`, `embedding` 인덱스, Row Level Security |
| 4 | 실거래가 데이터 적재 | 국토교통부 API 수집 데이터 → `houses_data` 적재 |
| 5 | 뉴스·집값 영향 요인 데이터 적재 | 개발2가 수집한 뉴스 데이터 → `news_signals` 적재 |
| 6 | 백엔드 연동 지원 | `SUPABASE_URL`, `SUPABASE_KEY` 공유, 쿼리 패턴 협의 |

#### C. 기획 업무

| # | 작업 | 상세 |
|---|------|------|
| 1 | PRD·로드맵 관리 | `01_PRD_HomeSignalAI.md` 유지보수, 기능 우선순위 재정립 |
| 2 | 미구현 기능 우선순위 | Must/Should/Nice 재정렬, 팀 리소스에 맞는 조정 |
| 3 | 사용자 시나리오 정의 | 예측 조회, 챗봇 질의, 뉴스 인사이트 시나리오 명세 |
| 4 | KPI·성공 지표 정의 | 예측 정확도, 응답 속도, 사용자 만족도 측정 방안 |
| 5 | 릴리스 계획 | 소프트 런칭 → 하드 런칭 일정 수립 |
| 6 | 사용자 리서치 (선택) | 동대문구 투자자·실거주자 대상 니즈 검증 |
| 7 | 프로젝트 총괄·디자이너 협업 | 화면 명세 수령, IA 기반 화면 정의서 전달 |

### 3.3 산출물

- 국토교통부 API 연동 코드 (수집 스크립트/서비스)
- Supabase 프로젝트 및 4개 테이블 (인덱스·RLS 포함)
- `houses_data`, `news_signals` 등 적재 데이터
- 업데이트된 PRD, 로드맵, 우선순위 문서
- 사용자 시나리오 및 KPI 정의서

### 3.4 협업 포인트

| 대상 | 협업 내용 |
|------|-----------|
| **개발2** | 테이블 스키마, 데이터 형식, 적재 타이밍 협의, 뉴스 데이터 수령 |
| **백엔드** | `forecast/service.py`에서 사용할 데이터 구조 전달, Supabase 연결 정보 공유 |
| **프로젝트 총괄** | PRD·화면 명세 수령 |
| **디자이너** | 기획 명세 기반 화면 설계 요청 |

## 4. 개발2 — 뉴스 크롤링 및 집값 영향 요인 데이터 담당

### 4.1 역할 정의

**뉴스 크롤링**과, 국토교통부 집값 데이터에 **영향을 주는 요인들**(뉴스, 정책, 이슈)의 **데이터 수집 및 모델**을 담당합니다. Supabase DB 운영은 개발1이 담당합니다.

### 4.2 담당 업무 상세

#### A. 뉴스 크롤링

| # | 작업 | 상세 |
|---|------|------|
| 1 | 구글 뉴스 크롤러 구현 | 동대문구·부동산 관련 키워드(GTX, 재개발, 청량리 등) 수집 |
| 2 | 키워드 목록 관리 | `03_AI_Model_Pipeline.md` 피처 정의에 맞는 키워드 확정 |
| 3 | 크롤링 스케줄 | 일별 또는 주별 수집, 중복 제거, 개발1(Supabase)에 데이터 전달 |
| 4 | 임베딩 생성 연동 | 뉴스 본문 → OpenAI `text-embedding-3-small` → embedding 포함 데이터 생성 |
| 5 | 크롤링 차단 대응 | robots.txt 준수, 요청 간격 조절, RSS·대체 소스 검토 |

#### B. 집값 영향 요인 데이터

| # | 작업 | 상세 |
|---|------|------|
| 1 | 집값 영향 요인 데이터 모델 설계 | 뉴스(`news_signals`), 이벤트 더미·정책 발표 등 스키마 제안 |
| 2 | 수집 데이터 형식 정의 | 개발1이 Supabase에 적재할 수 있는 JSON/CSV 등 형식 합의 |
| 3 | Vector DB 검색 로직 (선택) | `VectorDBInterface` ABC 준수, 백엔드와 쿼리 인터페이스 협의 |

### 4.3 집값 영향 요인 예시 (데이터에 반영할 항목)

| 요인 유형 | 예시 | 저장 대상 |
|-----------|------|-----------|
| **뉴스 키워드** | GTX, 재개발, 청량리, 이문휘경뉴타운, 금리, 분양 | `news_signals.keywords` |
| **정책 이벤트** | 전매 제한 해제, 금리 동결, 대규모 분양 | `news_signals` 메타데이터 또는 이벤트 테이블 |
| **시즌성** | 개학, 이사시즌, 결혼시즌 (달력 기반) | 별도 설정 또는 피처 테이블 |

### 4.4 산출물

- 뉴스 크롤러 코드 (Python 스크립트/서비스)
- 뉴스 데이터 (키워드, embedding 포함) — 개발1 Supabase 적재용
- 집값 영향 요인 데이터 모델·스키마 제안

### 4.5 협업 포인트

| 대상 | 협업 내용 |
|------|-----------|
| **개발1** | 테이블 스키마, 데이터 형식, 적재 형식 합의, 뉴스 데이터 전달 |
| **백엔드** | `shared/vector_db.py` Mock → 실제 구현체 시 쿼리 인터페이스 협의 |
| **개발1(기획)** | 뉴스 키워드 목록, 집값 영향 요인 우선순위 합의 |

## 5. 디자이너 — v0 AI 활용 UI/UX 설계

### 5.1 역할 정의

**v0 AI**를 활용하여 프론트엔드 **UI/UX 설계** 및 컴포넌트를 생성합니다.

### 5.2 활용 도구

| 항목 | 내용 |
|------|------|
| **v0 AI** | Vercel AI가 제공하는 프롬프트 기반 UI 컴포넌트 생성 도구 |
| **협업** | API 응답 구조(`07_Handover_Document.md` §6) 기반 설계, 프로젝트 총괄·백엔드와 응답 형식 사전 합의 |

### 5.3 담당 업무 상세

| # | 작업 | 상세 |
|---|------|------|
| 1 | 정보 구조(IA) 설계 | 메인 > 예측 / 챗봇 / 뉴스 인사이트 |
| 2 | v0 AI로 와이어프레임 | 핵심 3개 화면 + 에러/Fallback 상태 |
| 3 | UI 디자인 시스템 | 컬러, 타이포, 컴포넌트 규격 (톤앤매너: 신뢰감 블루 + 따뜻한 포인트) |
| 4 | 차트 시각화 설계 | 시계열 꺾은선, 신뢰구간, 키워드 빈도 바 차트 |
| 5 | 챗봇 인터페이스 | 메시지 버블, 로딩 상태, 피드백(👍/👎) UI |
| 6 | 반응형 설계 | 모바일 우선 또는 데스크톱 우선 결정 |
| 7 | 에러/Fallback 상태 | 네트워크 오류, AI 장애 시 "일시적 장애" 배너 |
| 8 | v0 AI 컴포넌트 생성 | 예측 카드, 챗봇 버블 등 프롬프트 작성·생성 |

### 5.4 v0 AI 활용 시 참고

1. **API 응답 구조 확인** — `07_Handover_Document.md` §6.2 시계열·챗봇·뉴스 응답 JSON 구조 참고
2. **컴포넌트 단위로 요청** — "예측 차트 카드", "출처 카드", "키워드 태그" 등 소규모 단위로 프롬프트 작성
3. **한국어·톤앤매너 명시** — "동대문구 부동산", "신뢰감 있는 전문가 톤", "30~50대 타겟" 등 포함

### 5.5 산출물

- IA 문서, v0 AI 기반 와이어프레임
- UI 디자인 시스템 (컬러, 타이포, 컴포넌트)
- v0 AI 생성 컴포넌트 (React/Next.js 등)
- 에러·Fallback 상태 디자인

### 5.6 협업 포인트

| 대상 | 협업 내용 |
|------|-----------|
| **프로젝트 총괄** | PRD·화면 명세 수령, 프론트엔드 구현 방향 합의 |
| **백엔드** | API 응답 구조 공유, 에러/로딩 상태 정의 |
| **개발2** | 데이터 시각화(차트)에 필요한 필드 확인 |
