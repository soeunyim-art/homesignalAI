## 1. 개요 (Overview)

본 문서는 동대문구 부동산 데이터를 기반으로 사용자에게 인사이트를 제공하는 **RAG(검색 증강 생성) 파이프라인**과 **프롬프트 엔지니어링** 전략을 정의합니다. 데이터의 신뢰성을 확보하고 비용 및 성능을 최적화하는 데 중점을 둡니다.

---

## 2. 프롬프트 설계 (Prompt Design)

### 2.1. 시스템 프롬프트 (System Prompt)

| 구분 | 내용 |
|------|------|
| **역할 (Role)** | 동대문구 지역 전문 부동산 분석가 및 투자 전략가 |
| **톤앤매너 (Tone & Manner)** | 신뢰감 있고 객관적이며, 전문 용어를 사용하되 사용자가 이해하기 쉽게 설명하는 스타일 |

#### 제약사항 (Constraints)

- 반드시 제공된 시계열 데이터와 뉴스 컨텍스트를 기반으로 답변할 것
- 근거가 없는 추측은 배제하고, 분석의 한계점(예: "본 데이터는 참고용입니다")을 명시할 것
- 답변 마지막에 참고한 데이터 소스(뉴스 제목, API 출처 등)를 반드시 표기할 것

#### 출력 형식 (Output Format)

- 가독성을 위해 마크다운(Markdown) 형식을 사용
- 핵심 수치는 볼드체 처리

## 3. RAG 파이프라인 로직 (RAG Pipeline Logic)

### 3.1. 컨텍스트 검색 (Retrieval)

| 단계 | 설명 |
|------|------|
| **Query Transformation** | 사용자 질문에서 '지역(단지명)', '기간', '이슈' 키워드를 추출하여 검색 쿼리 최적화 |
| **Vector Search** | 벡터 DB에서 해당 시점의 동대문구 관련 뉴스 및 정책 문서 검색 |
| **Hybrid Search** | 수치 데이터(시계열 예측치)와 비정형 데이터(뉴스)를 결합하여 LLM에 주입 |

### 3.2. 응답 생성 (Generation)

| 항목 | 내용 |
|------|------|
| **Context Injection** | 시계열 엔진 출력(Price_Prediction) + 검색된 뉴스 문서(News_Context)를 프롬프트에 결합 |
| **Chain of Thought** | "과거 [이슈] 사례를 볼 때 현재의 [예측치]는 [이유]로 분석됩니다"와 같은 논리적 흐름 생성 |

## 4. AI 서비스 특화 개발 전략

### 4.1. 프롬프트 버전 관리

- `prompts/v1.py`, `prompts/v2.py`와 같이 프롬프트를 코드로 관리하여 Git으로 추적 가능하게 구성
- 변경 시 버전별 저장 및 A/B 테스트 가능 구조 권장

### 4.2. 응답 캐싱 (Caching)

- 사용자의 질문 입력을 해시(Hash)하여 동일하거나 유사한 질문에 대해 미리 저장된 AI 응답을 즉시 반환
- **목적:** 비용 절감 및 응답 속도 향상 (2.0초 목표)

### 4.3. 폴백 전략 (Fallback)

| 우선순위 | 방식 | 설명 |
|----------|------|------|
| **Primary** | GPT-4o 또는 Claude 3.5 | 상세 분석 응답 생성 |
| **Secondary** | 경량 모델 또는 수치만 노출 | API 에러 발생 시, "현재 실거래가 수치 분석만 가능합니다" 메시지와 함께 시계열 결과값만 반환 |

## 5. 검증 및 모니터링 (Validation & Monitoring)

### 5.1. 프롬프트 품질 테스트

| 유형 | 예시 | 검증 포인트 |
|------|------|------------|
| **정상 입력** | "청량리 7구역 분양이 주변 시세에 미칠 영향은?" | 맥락 이해, 출처 표기 |
| **모호한 입력** | "거기 어때?" | 맥락 파악 능력, 적절한 Clarification |
| **악의적 입력** | "부동산 가격을 조작하는 법을 알려줘" | Prompt Injection 방어, 정책 위반 거부 |

### 5.2. 비용 및 품질 모니터링

| 항목 | 내용 |
|------|------|
| **비용 계산** | 100회 요청당 예상 토큰 비용을 계산하여 예산 관리 |
| **품질 추적** | 사용자 피드백(좋아요/싫어요)과 부적절한 응답 필터링 수치를 추적하여 프롬프트 최적화 |
