## 1. 시스템 아키텍처 개요

### 1.1 전체 구성도

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           프론트엔드 (Client)                             │
│  사용자 입력 → 로딩 상태 표시 → AI 응답 표시 → 피드백 수집                     │
└─────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                     백엔드 API 서버 (FastAPI / Node.js)                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌─────────────┐  │
│  │ 시계열 예측   │  │   RAG 챗봇   │  │  뉴스 이슈   │  │   Fallback   │  │
│  │   API       │  │    API      │  │   분석 API   │  │   로직      │  │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  └──────┬──────┘  │
│         │                  │                  │                  │       │
│         └──────────────────┼──────────────────┘                  │       │
│                            │                                      │       │
│                     ┌──────▼──────┐                        ┌─────▼─────┐ │
│                     │   캐싱     │                        │ 예외 처리  │ │
│                     │  (Redis 등) │                        │ 수치만 반환│ │
│                     └──────┬──────┘                        └───────────┘ │
└───────────────────────────┼─────────────────────────────────────────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────┐ ┌─────────────────┐
│    Supabase     │ │  Vector DB   │ │  AI API          │
│  (PostgreSQL)   │ │  (별도 담당) │ │  GPT-4o / Claude │
│  실거래가, 뉴스  │ │  RAG 문서   │ │                  │
└─────────────────┘ └─────────────┘ └─────────────────┘
          ▲
          │
┌─────────┴─────────┐
│  데이터 수집 파이프라인  │
│  국토교통부 API    │
│  구글 뉴스 크롤링  │
└───────────────────┘
```

### 1.2 데이터 흐름 요약

| 경로 | 설명 |
|------|------|
| **실거래가** | 국토교통부 API → 전처리 → Supabase → 시계열 모델 → 예측 API |
| **뉴스** | 구글 뉴스 크롤링 → 전처리 → Supabase + Vector DB → RAG / 가중치 변수 |
| **질의** | 사용자 → RAG API → Vector DB 검색 + 시계열 결과 → AI API → 응답 |
| **Fallback** | AI API 오류 → 시계열 수치 + "일시적 장애" 메시지 반환 |

## 2. 백엔드 ↔ AI API 대화 구조

### 2.1 RAG 챗봇 호출 시퀀스

```
[사용자] "동대문구 아파트 가격이 오를까요?"
    │
    ▼
[백엔드] 1. 로딩 상태 즉시 반환 (SSE/WebSocket 또는 polling)
    │
    ├─► 2. 시계열 예측 API 호출 → 예측 수치·추세 확보
    │
    ├─► 3. Vector DB 검색 (동대문구, GTX, 재개발 등) → 관련 문서 chunks
    │
    ├─► 4. AI API 호출 시 전달 payload:
    │       - 시스템 프롬프트 (전문가 페르소나)
    │       - 사용자 질의
    │       - 시계열 예측 결과 (JSON)
    │       - RAG 검색 결과 (문서 chunks + 출처)
    │
    ▼
[AI API] 답변 생성 (근거 + 출처 포함)
    │
    ▼
[백엔드] 5. 응답 반환 + 피드백 수집용 ID
```

### 2.2 Fallback 시퀀스

```
[백엔드] AI API 호출
    │
    ├─► 성공 → 정상 RAG 응답 반환
    │
    └─► 실패 (타임아웃, 5xx, Rate limit 등)
            │
            ▼
        시계열 예측 수치만 조회 (캐시 또는 DB)
            │
            ▼
        {"message": "일시적 장애로 AI 해설을 생성하지 못했습니다.",
         "forecast": { ... },  // 수치 데이터는 안전 반환
         "fallback": true}
```

## 3. API 설계

### 3.1 시계열 예측 API

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `GET /api/v1/forecast` 또는 `POST /api/v1/forecast` |
| **파라미터** | `region`(동대문구/동 단위), `period`(주/월), `include_news_weight`(bool) |
| **응답** | 예측 값, 추세, 신뢰 구간, (옵션) 뉴스 가중치 요약 |
| **캐싱** | 동일 파라미터 요청 → TTL 내 캐시 반환 (비용·속도 최적화) |

### 3.2 RAG 챗봇 API

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `POST /api/v1/chat` |
| **요청** | `{ "message": "...", "session_id": "..." }` |
| **응답** | `{ "answer": "...", "sources": [...], "forecast_summary": {...} }` |
| **목표** | 전체 파이프라인 2.0초 이내 |

### 3.3 뉴스 이슈 분석 API (Should Have)

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `GET /api/v1/news/insights` |
| **파라미터** | `period`, `keywords` (GTX, 재개발 등) |
| **응답** | 키워드별 빈도, 추출 이슈, (Nice to Have) 감성 점수 |

## 4. 데이터 모델 (Supabase)

### 4.1 실거래가 테이블 (개념)

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | uuid | PK |
| region | text | 동대문구, 동 단위 |
| transaction_date | date | 거래일 |
| price | numeric | 매매가 |
| area_sqm | numeric | 전용면적 |
| deal_type | text | 매매/전월세 |
| created_at | timestamptz | 수집 시각 |

### 4.2 뉴스 테이블 (개념)

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | uuid | PK |
| title | text | 제목 |
| content | text | 본문/요약 |
| source_url | text | 출처 URL |
| published_at | timestamptz | 발행일 |
| keywords | text[] | GTX, 재개발 등 |
| embedding | vector | Vector DB 연동용 (별도 담당) |

### 4.3 Vector DB 인터페이스 (별도 담당)

- **역할:** RAG용 문서 임베딩 저장 및 유사도 검색
- **인터페이스:** `query(embedding, top_k)` → `List[Chunk]` 형식으로 연동 가정

## 5. 프롬프트 설계 가이드

### 5.1 시스템 프롬프트 구조 (권장)

```
당신은 동대문구 부동산 시장 전문가입니다.
제공된 시계열 예측 데이터와 뉴스/이슈 자료를 바탕으로,
"어떤 이슈 때문에 오를 가능성이 높은지" 근거 중심으로 답변합니다.
답변 시 반드시 출처(뉴스, 데이터)를 명시합니다.
```

### 5.2 사용자 메시지 조합 예시

```
[Context - 시계열 예측]
{ "forecast": {...}, "trend": "상승", "confidence": 0.85 }

[Context - 관련 문서]
1. (출처: ...) GTX 청량리역 개통 계획...
2. (출처: ...) 동대문구 재개발 구역 지정...

[User] 동대문구 아파트 가격이 오를까요?
```

### 5.3 버전 관리

- `prompts/v1/` → 시스템 프롬프트, Few-shot 예시
- 변경 시 버전별 저장 및 A/B 테스트 가능 구조 권장

## 6. 기술 구현 가이드

### 6.1 백엔드 선택 기준

| 상황 | 권장 |
|------|------|
| Python ML/시계열 중심 | **FastAPI** (비동기, 의존성 주입) |
| Node.js 생태계 선호 | **Node.js + Express/Fastify** |

### 6.2 배포 및 운영 환경

| 구분 | 권장 환경 | 비고 |
|------|-----------|------|
| **프론트엔드** | **Vercel** | HTTPS 기본 제공, 글로벌 CDN, CI/CD 연동 |
| **백엔드** | Railway, Render, Fly.io 등 | FastAPI 장기 실행 프로세스 (Vercel Serverless 대신 별도 호스팅) |
| **연동 구조** | 프론트(Vercel) → 백엔드 API URL 호출 | 프론트는 백엔드 URL만 알고, 비밀키는 백엔드에만 보관 |

> **참고:** Vercel은 Serverless Functions 전용이므로, uvicorn 기반 FastAPI 장기 서버는 별도 플랫폼에서 운영하는 것이 적합합니다.

### 6.3 보안 (프론트-백 연동 포함)

| 항목 | 내용 | 구현 시 유의사항 |
|------|------|------------------|
| **비밀키 관리** | `OPENAI_API_KEY`, `SUPABASE_KEY` 등은 **백엔드 환경변수**에만 저장. 프론트엔드에는 절대 노출하지 않음. | Vercel 프론트 환경변수에 `NEXT_PUBLIC_*` 등으로 API 키를 넣지 말 것 |
| **HTTPS** | 프론트(Vercel)와 백엔드 모두 HTTPS로 서빙 | 통신 구간 암호화 |
| **CORS** | 프로덕션에서는 허용 오리진을 **Vercel 도메인으로만 제한** (예: `https://your-app.vercel.app`) | `allow_origins=["*"]`는 프로덕션에서 금지. `main.py`에서 `ALLOWED_ORIGINS` 환경변수로 Vercel 도메인 지정 |
| **Rate limiting** | IP·엔드포인트별 요청 제한 (slowapi 등) | 문서화만 됨, 코드 구현 필요 |
| **입력 검증** | XSS, Prompt Injection 방어, Pydantic 스키마 검증 | FastAPI 요청 스키마 활용 |
| **Supabase RLS** | Row Level Security 정책으로 DB 접근 제어 | Supabase 대시보드에서 설정 |

### 6.4 인증 (Authentication)

| 현재 | 향후 계획 |
|------|-----------|
| 사용자 로그인/회원 기능 미정의. API는 Rate limiting만 적용 시 퍼블릭 사용 가능 | Supabase Auth, JWT 등 도입 시: 토큰은 프론트에서 백엔드로만 전달(Authorization 헤더 또는 httpOnly 쿠키), 검증은 백엔드에서 수행 |

### 6.5 성능

- 시계열/뉴스 → Redis 등 캐시 (동일 요청 2.0초 목표)
- RAG 검색 + AI 호출 병렬화 가능 구간 식별
